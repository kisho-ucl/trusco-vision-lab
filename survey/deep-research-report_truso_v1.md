# 物流倉庫における複数天井カメラ統合 top-view と人物軌跡に基づく作業認識の関連研究調査

## A. 結論

結論から言うと、**ご提示の設定に「非常に近い」研究はありますが、端から端まで一致する研究はかなり限られます**。特に近いのは、物流倉庫の天井に多数の固定カメラを設置し、歪み補正・位置合わせ・統合座標化を行って、倉庫全体で作業者を追跡する研究です。代表例として、Moriらの 2025 年 arXiv 論文は、**物流倉庫の天井に設置した 19 台の広角カメラ**から、床面基準の整列と外観特徴の活用により、**倉庫全体で作業者を追跡する**ことを扱っています。もう一つ近いのは Cheng らの 2025 年の warehouse monitoring / digital twin 系論文で、**ceiling-mounted cameras、物体追跡、knowledge-graph based world model**を組み合わせた**倉庫監視とデジタルツイン**を扱っています。ですが、これらは主に**追跡・位置可視化・資産追跡・scene analysis**までであり、**各 worker ID に対して長時間軌跡から task label を推定する**ところまでは踏み込んでいません。つまり、**tracking 基盤に近い研究はあるが、tracking から per-worker task recognition へ接続する研究はまだ薄い**、というのが最も重要な結論です。 citeturn7view1turn12search1

一方で、**物流・工場の作業認識そのもの**は既に研究されています。ただしその主流は、**倉庫全体の top-view 監視**ではなく、**wearable / IMU / IoT / motion capture / workstation camera / depth camera / LiDAR** です。OpenPack は、物流 packaging work を対象に、**IMU、depth、LiDAR、IoT device readings** を含む 53.8 時間超・16 名の大規模 multimodal dataset を公開しており、LARa は picking / packing シナリオを **OMoCap・IMU・RGB camera** で記録した logistics HAR dataset です。InHARD も industrial collaborative robotics 文脈の工場作業データセットですが、こちらも **RGB + skeleton / workstation-scale** が中心です。したがって、**task recognition はあるが、視点と空間スケールが違う**、というズレがあります。 citeturn13view0turn13view2turn14search4turn28search3

建設分野はかなり状況が異なります。こちらでは、**site surveillance videos を用いた automated work sampling、worker activity recognition、pose-based analysis、multi-camera pose fusion** が比較的豊富に確認できます。Luoらの 2018 年 Automation in Construction 論文は site surveillance videos に two-stream CNN を適用して automated work sampling を行い、Robertsらの 2020 年論文は**2D pose estimation + tracking** により construction worker の per-frame activity analysis を行っています。さらに Jangらの 2023 年論文は construction human-robot collaboration で**multi-camera** から **3D pose** を推定して activity recognition を行っています。したがって、**「俯瞰／遠景監視カメラから worker activity を読む」研究の厚みは、物流・倉庫より建設の方が明らかに厚い**と評価してよいです。 citeturn31search0turn43search4turn29view0turn32view0

したがって、研究新規性候補 A〜D の評価は、次のように整理できます。**Candidate claim A は概ね妥当**です。公開文献として確認できた近接例は tracking / monitoring 側に偏っており、**multi-camera top-view warehouse tracking から per-worker task recognition までを一気通貫で扱う研究**は、この探索範囲ではほとんど確認できませんでした。**Candidate claim B もかなり妥当**です。construction では surveillance / work sampling 系が豊富である一方、logistics / factory では LARa、OpenPack、InHARD のように**wearable / workcell / RGB-D / MoCap 側**が主流です。**Candidate claim C は強く支持**されます。trajectory-only の industrial behavior 論文が主に walking / loitering / stillness のような**粗い移動パターン**を扱うのに対し、LART や RetailAction、construction の object-orientation-ReID 系研究は、**作業意味の識別に context と interaction が必要**であることを示しています。**Candidate claim D は有望だが、文献上はまだ「完全実証」ではない**、というのが妥当です。OpenPack は IoT readings が特定作業と強く結びつくことを示し、MoIL は limited labels 下で industrial HAR を改善していますが、**warehouse-wide top-view に worker-object-zone relation・trajectory・ReID・IMU weak labels を統合した研究**はまだ見当たりません。 citeturn7view1turn12search1turn13view0turn14search4turn28search3turn31search0turn43search4turn40search1turn41view0turn34view1turn36search1

この結論には一つ留保があります。物流倉庫の実応用は企業内 PoC や非公開実装が多く、公開論文・公開データセットだけでは全体像を過小評価している可能性があります。ただし、**公開・検証可能な研究としては、top-view multi-camera warehouse tracking と per-worker task recognition の間にまだ大きな空白がある**、という判断はかなり堅いです。 citeturn13view0turn23view0turn32view0

## D. Warehouse / logistics に絞った結果

warehouse / logistics に限定すると、現時点で見つかったものは大きく三群に分かれます。第一に、**ceiling camera からの tracking / monitoring / digital twin** で、Mori 2025 と Cheng 2025 が代表です。第二に、**packaging / picking / packing station の multimodal HAR** で、LARa、OpenPack、MoIL が代表です。第三に、**trajectory / localization data を process model や simulation に接続する研究**で、Aslan 2023 や warehouse digital twin 系論文が近い位置にあります。つまり、**「tracking はある」「task recognition はある」「digital twin 接続もある」**のですが、これらが**同じ論文の中で end-to-end に統合されていない**のが現状です。 citeturn7view1turn12search1turn13view0turn14search4turn36search1turn25search0turn26search0

逆に、warehouse / logistics で**見つからなかったもの**も明確です。今回確認できた公開文献の範囲では、**多数の ceiling camera を統合した warehouse-wide stitched top-view** の上で、**worker ID ごとの長時間 trajectory を抽出し、その worker に対して Inspect / Sort / Transport のような作業ラベルを長時間・低アノテーションで推定する公開研究**は確認できませんでした。近い論文は tracking まで、あるいは workstation-local な作業認識までで止まっています。これは、そのまま Candidate claim A の根拠になります。 citeturn7view1turn12search1turn13view0turn14search4turn28search3

この差が construction と warehouse / factory の間で生じやすい理由も、文献からかなり説明できます。construction では、Sherafat らの review が示すように、**location tracking → activity recognition → activity tracking → performance monitoring** という系統が早くから意識されており、work sampling や productivity monitoring の用途が明確です。Luo 2018 のように**productive / semi-productive / non-productive** のような粗い作業カテゴリは site surveillance と相性がよく、Roberts 2020 のように body posture を使えば per-frame activity analysis に踏み込めます。また、Bird’s-Eye View camera で construction worker を tracking する論文もあります。つまり construction は、**現場全体を離れた地点から監視する必然性**と、**粗い生産性分析の需要**が surveillance 研究を後押ししています。 citeturn32view0turn31search0turn43search4turn44search0

これに対して物流倉庫・工場では、作業意味が**人の移動そのものではなく、手元の物体、注文書、棚、台車、スキャナ、梱包対象、棚状態**に強く依存します。OpenPack はまさに、IoT devices の reading が特定作業に強く関連することを強調しており、RetailAction も「人の box を追う」より**interaction point**を注釈すべきだと主張しています。LART も、pose だけではなく**appearance / scene context** が必要な action があることを明示しています。つまり、warehouse では**同じ位置・同じ姿勢でも作業意味が変わりやすい**ため、construction より top-view のみで fine-grained task を読むことが難しいのです。 citeturn13view0turn34view1turn41view0

さらに、top-view / ceiling camera 固有の技術課題も強いです。top-view review は privacy や reduced occlusion を利点として挙げる一方で、**restricted field of view、unique perspective、standardized dynamic datasets の不足**を指摘しています。overhead fisheye review は、**単一カメラでは大空間を十分にカバーできず、複数 overhead fisheye camera を使うと camera overlap と people counting / tracking ambiguity が生じる**こと、そして**fisheye cameras 向け ReID 手法はまだ非常に少ない**ことを述べています。これは、顔が見えず、手元も弱く、作業者の服装が似ており、遮蔽・交差・退出再入場が多い倉庫にそのまま当てはまります。 citeturn20search6turn23view0

warehouse / logistics の公開研究が少ない理由としては、**データ公開の難しさ**も大きいと考えられます。OpenPack は industrial domain dataset が少ない理由として、**実環境との密接な連携、作業手順書に基づくラベル定義、経験者の被験者確保**を挙げています。倉庫ではこれに加えて、商品情報、注文情報、棚配置、運用ノウハウ、従業員プライバシーが映像に含まれやすいので、construction site の surveillance movie より**公開しにくい**と考えるのが自然です。これは文献に明示的に数量比較された事実ではありませんが、OpenPack の dataset construction の難しさと、公開 top-view warehouse dataset の希少さを見る限り、かなり説得力のある解釈です。 citeturn13view0turn13view2

ご研究設定で最も現実的な starting point は、**tracking-only**, **trajectory+zone**, **pose+tracklet**, **worker-object-zone relation** の四段階比較です。理由は明快で、Mori 2025 型の tracking は既に近い先行があり、Aslan 2023 は trajectory を process model / simulation に接続できることを示しており、construction と retail の先行研究は**姿勢・orientation・object interaction** を足すことで coarse motion を意味ある action に近づけられることを示しているからです。 citeturn7view1turn25search0turn43search4turn19search0turn34view1

特にご質問の「trajectory だけで何ができそうか」に対しては、**Transport は比較的扱いやすい**と見るのが妥当です。理由は、trajectory-only の industrial behavior 認識が walking / loitering / prolonged stillness / abnormal acceleration のような**運動学的に特徴のある class**を高精度に扱えているからです。これに対し **Inspect と Sort は trajectory だけでは曖昧**になりやすいです。LART は pose だけでなく contextualized appearance を必要とする action があることを示し、RetailAction は人 box ではなく interaction point 自体を注釈対象にしています。つまり、**Inspect / Sort は zone、worker orientation、近接 object、台車や商品との relation を足して初めて扱いやすくなる**、という見立てが文献的にも自然です。これは厳密には推論ですが、かなり強い推論です。 citeturn40search1turn41view0turn34view1

low annotation の観点では、**industrial-specific self-supervision と cross-modal weak labels の組合せ**が最も有望です。MoIL は OpenPack 系の complex work recognition で limited labels 下の改善を報告しており、OpenPack 自体が IoT readings を強い補助情報として位置付けています。また、一般 video 学習では UntrimmedNet が weak labels、VideoMAE が self-supervised pretraining、ActionCLIP / CLIP が zero-shot / few-shot transfer を支えます。したがって、**IMU や scanner 由来の weak labels を teacher とし、映像側では tracklet / zone / object relation を student とする構成**は、現在の公開文献の流れにかなり整合的です。 citeturn36search1turn13view0turn42search0turn42search1turn42search2turn42search3

## F. 修士研究として有望な Research Question

**RQ: trajectory と zone transition だけで、どこまで warehouse task を識別できるか。** 背景として、物流倉庫の multi-camera tracking と、worker position から process model / simulation を作る研究は存在しますが、その間の task labeling は薄いです。差分は、**global top-view trajectories を event / task に変換する mapping を定量評価すること**にあります。実験としては、Transport / non-Transport から始めて、次に Inspect / Sort を足し、trajectory-only、trajectory+zone、trajectory+zone+duration model を比較するとよいです。失敗しても、**「どの task が trajectory だけでは識別不可能か」**という設計知見が得られます。 citeturn7view1turn25search0

**RQ: Inspect と Sort の識別に、worker orientation と object / zone context はどれだけ効くか。** 背景として、construction では body posture、attention orientation、object detection、ReID の併用が activity recognition に効いており、retail では interaction point が中心的な表現になっています。差分は、これを**倉庫 top-view の worker-object-zone relation**に落とし込む点です。実験としては、trajectory-only に対して、head/body orientation、inspection table、sorting shelf、cart proximity、document / package detection を段階的に追加します。失敗しても、**どの context が本当に必要で、どの context が不要か**がわかります。 citeturn43search4turn19search0turn34view1

**RQ: IMU / smartphone / IoT 由来の weak labels は、top-view 映像モデルの教師として使えるか。** 背景として、OpenPack は IoT readings を有効な補助情報として扱い、MoIL は limited-label industrial HAR に効果を示しています。差分は、**wearable-centric label source を warehouse-wide vision へ蒸留する cross-modal weak supervision**です。実験では、一部作業者にだけ IMU / smartphone を持ってもらい、その task estimation を confidence-aware pseudo-label として camera-side temporal model に与え、純粋手ラベル学習と比較します。失敗しても、**どの task で cross-modal teacher が効くか、label noise をどこまで許容できるか**が分かります。 citeturn13view0turn36search1

**RQ: long-term per-worker task recognition において、minute-scale の temporal model はどれだけ重要か。** 背景として、現場作業は数分〜数十分継続し、role switching もあります。UntrimmedNet は untrimmed video の weakly supervised action detection を扱い、process mining は continuous stream を event log に変換する必要があります。差分は、**frame label ではなく segment label / transition model を主役にすること**です。実験としては、clip classifier 単体と、HMM / HSMM / temporal transformer を比較し、label sparsity に対する頑健性も評価します。失敗しても、**現実的な annotation unit は frame ではなく segment である**ことを定量的に示せます。 citeturn42search0turn24search7turn25search0

**RQ: task-level semantics を付加した digital twin は、trajectory-only twin よりも layout / staffing optimization に有利か。** 背景として、倉庫 monitoring / digital twin と、worker position からの simulation は既にありますが、task semantics を含む worker model は薄いです。差分は、**「どこを通ったか」ではなく「何の作業をしながら通ったか」**を twin に入れることです。実験では、trajectory-only twin と task-aware twin を比較し、レイアウト変更や staffing 変更に対する throughput・walking distance・inspection delay などの予測精度を評価します。失敗しても、**task recognition を入れる価値が本当にあるのか**を定量的に判定できます。 citeturn12search1turn25search0turn26search0
