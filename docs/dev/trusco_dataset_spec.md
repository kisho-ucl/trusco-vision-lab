# TRUSCO Dataset & Tracking Result Specification

このドキュメントでは、TRUSCO実倉庫データを用いた各種要素開発において、共通で利用されるトラッキング結果データ（JSON）、元のカメラ動画ファイル、およびそれらを実時刻で紐付けるタイムスタンプキャッシュ（ts_cache）の基本構造と配置ルールを定義します。

---

## 1. トラッキング結果データ (JSON)

### 1.1 物理配置
トラッキング処理によって出力される結果ファイルは、以下のディレクトリ構成で配置されます。
* **配置パス**: `dataset/fastops/result/<Result_ID>/track/modified_kalman_dtw2_adjusted_tracking_result.json`
* **実データリンク**: プロジェクトルートの `dataset/` は、共有マウント先（`/mnt/bigdata/01_projects/2024_trusco`）へのシンボリックリンクです。

### 1.2 ファイル構造と主要キー
全体はフレームごとのリスト（配列）構造になっています。

```json
[
  {
    "frame_id": 0,
    "tracks": [
      {
        "track_id": 148,
        "src_cam": "A1",
        "bbox": [
          6032.729012073499,
          843.2942788254944,
          240.0783689654436,
          160.03035410152313
        ],
        "original_bbox": [
          160.07403564453125,
          102.95695495605469,
          320.1480712890625,
          205.91390991210938
        ],
        "src_cam_duplicate_list": [
          "A1"
        ],
        "bbox_list": [
          [
            160.07403564453125,
            102.95695495605469,
            320.1480712890625,
            205.91390991210938
          ]
        ]
      }
    ]
  }
]
```

#### 主要項目:
* **`frame_id`**: 0から始まる処理フレームの連番。後述の通り、元の動画のフレームから偶数フレームが間引かれています。
* **`track_id`**: 追跡対象（人物等）の一意なID。※別スクリプト等で `subj_id` (被験者ID) が付与されている場合はそちらが優先されますが、トラッキング出力単体では `track_id` をベースに処理します。
* **`src_cam`**: 代表カメラのID（例: `"A1"`, `"B7"`）。
* **`src_cam_duplicate_list`**: 該当人物が同時に映り込んでいるカメラのリスト。

### 1.3 座標系と切り出し時の注意点
* **広域 (global) 座標系 (`"bbox"`)**:
  スティッチングされた巨大なマップ上の座標 `[cx, cy, w, h]` です。
  元画像（カメラ画像内）のローカル座標に戻すには、基準点 `BASE_X = 3885.356`, `BASE_Y = 812.703` を減算する必要があります。
  * `cx_local = cx_global - BASE_X`
  * `cy_local = cy_global - BASE_Y`
* **ローカル (local) 座標系 (`"original_bbox"`, `"bbox_list"`)**:
  各カメラの動画内での座標 `[cx, cy, w, h]` です。
  **動画から人物のクロップ画像を切り出す場合は、この `original_bbox` もしくは `bbox_list` 内の座標をそのまま直接使用します。**

## 2. 元カメラ動画ファイル

### 2.1 物理配置
各カメラが録画した、歪み補正済み（undistorted）の動画は以下のマウントディレクトリ配下に配置されます。
* **配置パス**: `/mnt/gazania/trusco-undis/2024/{date}/camera{cam_id}/video_??-??-??_{vid_idx}.mp4`
  * 例: `cameraA10` のディレクトリ配下に `video_10-00-00_00.mp4`, `video_10-00-00_01.mp4` ... のように時間帯ごとに分割して保存されます。
  * **注意**: `/mnt/gazania` は **Read-Only** です。ファイルの追加・編集は行わないでください。

## 3. タイムスタンプキャッシュ (ts_cache)

JSONの `frame_id` から実動画上の正しいファイル名およびフレーム位置を逆引きするため、タイムスタンプキャッシュを使用します。

### 3.1 物理配置
* **配置パス**: `dataset/ts_result/{date}-tscaches_adjusted.pkl` (pickle形式)
  * 例: 2024年10月3日の実験データであれば、`dataset/ts_result/20241003-tscaches_adjusted.pkl` を使用します。

### 3.2 データ構造 (Pickleの中身)
ロードされた `ts_cache` オブジェクトは、通常以下の要素を持つリスト（タプル）です。
1. **`ts_cache[0]` (フレーム対応マップ)**:
   * 辞書型。キーはカメラID（例: `"A1"` 等）、値はフレーム対応リスト。
   * インデックス（1秒間に5フレーム = 0.2秒刻みの経過時間に対応）を指定すると、動画ファイルとフレームのペア `(vid_idx, frm_idx)` が得られます。
2. **`ts_cache[1]` (カメラ別時間オフセット)**:
   * 辞書型。キーはカメラID、値は各カメラの動画の開始時間基準点（秒）。

### 3.3 フレーム特定ロジック
JSONの `frame_id` に対応する動画フレームは以下の手順で特定します。

1. **実フレーム番号 (`orig_frame`) への復元**:
   トラッキング結果は偶数フレームが間引かれているため、元のフレーム番号を以下で算出します。
   $$\text{orig\_frame} = \text{frame\_id} \times 2 + 1$$
2. **実時間 (秒) の算出**:
   1フレームあたり 0.2 秒に相当します。処理の開始時刻（例: `"10:00:00"`) からの経過秒数を足します。
   $$\text{cur\_in\_sec} = \text{START\_TIME\_in\_seconds} + (\text{orig\_frame} \times 0.2)$$
3. **インデックスの算出と逆引き**:
   カメラごとの開始時間オフセット `ts_cache[1][cam]` との差分から、`ts_cache[0]` のインデックス `idx` を算出します。
   $$\text{idx} = \text{int}(5 \times (\text{cur\_in\_sec} - \text{ts\_cache}[1][\text{cam}]))$$
   得られた `idx` を用いて、`ts_cache[0][cam][idx]` から `(vid_idx, frm_idx)` を取得します。
4. **動画ファイルのロード**:
   上記で取得した `vid_idx` に対応する動画ファイル（例: `video_??-??-??_{vid_idx:02d}.mp4`）を開き、`frm_idx` 番目のフレームを読み出します。

## 4. Result ID と実験日時のマッピング（設計課題）

### 4.1 課題
トラッキング結果のディレクトリ名である `Result_ID` (例: `78`) と、元データの実証実験日時（例: `20241003_1000_1100`）の対応関係は、物理データ（`/mnt` 側）のディレクトリ内には記録されていません。

### 4.2 解決方針
共有マウント先である `/mnt` 領域は **Read-Only** であるため、開発するアプリケーション側（リポジトリ内）に設定ファイル `config/dataset_mapping.json` などを保持し、システム的にマッピングを静的定義して解決します。

* **マッピング定義の例 (`config/dataset_mapping.json` 等)**:
```json
{
  "78": {
    "session_name": "kisho_20241003_1000_1100",
    "date": "2024-10-03",
    "start_time": "10:00:00",
    "ts_cache_file": "dataset/ts_result/20241003-tscaches_adjusted.pkl",
    "undis_dir": "/mnt/gazania/trusco-undis/2024/2024-10-03"
  }
}
```
各要素技術（アノテーションツール、切り出しスクリプトなど）を動かす際は、このマッピング定義を仲介して、Result IDから正しい日付、開始時間、および `ts_cache_file` のパスを自動参照する仕組みを構築します。
